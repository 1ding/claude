# 1 总则

+   文档编号：S00
+   文档名称：软件研发文档通用结构（High-Level Structure）
+   目的：定义软件研发过程文档的通用结构框架，提高各阶段规范文档的一致性和可维护性
+   范围：适用于软件研发过程中所有阶段性规范文档的结构设计与编写
+   管理要点：文档结构框架、章节组织规范、内容要素框架
+   文档密级：【商密二级】 -- 外发必追责。

# 2 HSL框架综述

## 2.1 HSL的定位与作用

+   HSL（High-Level Structure，高阶结构）是软件研发文档体系的元规范，为各阶段规范文档提供统一的结构框架和编写指引。
+   HSL不直接约束最终交付文档（如战略规划、宏观需求等），而是约束规范文档本身的编写方式，确保规范文档之间具有一致的组织逻辑和表达风格。
+   HSL与ISO/IEC导则的关系：参考ISO/IEC导则第1部分附件SL（High Level Structure）的理念，但针对软件研发文档特点进行裁剪和适配，不追求ISO标准的完整性和严格性。

## 2.2 HSL框架的层次结构

+   HSL框架分为三个层次，从宏观到微观依次为：

    | 层次 | 名称         | 作用                             | 对应章节 |
    | ---- | ------------ | -------------------------------- | -------- |
    | L1   | 文档结构框架 | 定义文档的整体组织方式和基本属性 | 第3章    |
    | L2   | 章节组织规范 | 定义章节的层次结构和编排逻辑     | 第4章    |
    | L3   | 内容要素框架 | 定义各类章节应包含的核心要素     | 第5章    |

    +   HSL框架层次结构

## 2.3 HSL的适用范围与裁剪原则

+   适用范围：
    -   软件研发过程规范文档
    -   其他需要高度结构化的技术规范文档
    -   不适用于最终交付文档（如项目文档、设计文档等），这些文档由各自的规范文档约束

+   裁剪原则：
    -   各规范文档可根据自身特点裁剪HSL框架，但应在规范文档中明确说明裁剪理由
    -   L1（文档结构框架）为强制层次，不可裁剪
    -   L2-L3可根据文档复杂度和目标受众进行适度裁剪
    -   裁剪后应保持文档的完整性和可理解性

## 2.4 HSL与其它规范的关系

+   HSL与C01（内容表达规范）、C02（排版格式规范）的关系：
    -   C01和C02是通用规范，适用于所有类型文档
    -   HSL是结构规范，专门针对规范文档本身
    -   三者关系：HSL定义"写什么"（结构和要素），C01定义"怎么写"（表达方式），C02定义"怎么排版"（格式样式）
    -   优先级：HSL > C01 > C02（当三者冲突时，HSL优先）

+   HSL与阶段规范文档的关系：
    -   各阶段规范文档是HSL的实例化应用，每个规范文档都应遵循HSL框架
    -   各阶段规范文档可在HSL框架基础上增加特定领域的要求，但不应违背HSL的核心原则
    -   HSL的演进应基于各阶段规范文档的实践反馈，保持框架的实用性和适应性

# 3 文档结构框架（L1）

## 3.1 文档结构框架的作用

+   文档结构框架定义规范文档的整体组织方式和基本属性，确保各规范文档具有一致的宏观结构和可识别性。
+   结构框架包含两个方面：
    -   文档的基本属性和标识信息（元数据）
    -   文档的整体组织方式（三部分结构）

## 3.2 标准文档结构

+   所有规范文档应遵循以下标准结构：

    ```text
    规范文档标准结构
    ├── 前置部分
    │   ├── 第1章 总则（强制）
    │   └── 第2章 {规范对象}综述（强制）
    ├── 正文部分
    │   ├── 第3章 {核心内容1}（强制）
    │   ├── 第4章 {核心内容2}（强制）
    │   ├── ...
    │   └── 第N章 {核心内容N-2}（强制）
    └── 附录部分
        ├── 附录A（资料性）术语表（推荐）
        ├── 附录B（资料性）{其他参考资料}（可选）
        └── 修订记录（强制）
    ```

+   结构说明：
    -   前置部分固定为2章，不可增减
    -   正文部分章节数量根据规范对象的复杂度确定，通常为5-10章
    -   附录部分至少包含修订记录，术语表为推荐项
    -   章节编号连续，不得跳号

## 3.3 前置部分的结构要求

### 3.3.1 第1章：总则

+   章节定位：文档的元数据声明章节，提供文档的基本识别信息。
+   必须包含的元数据：

    | 元数据项 | 格式                    | 说明                                                            | 示例                                            |
    | -------- | ----------------------- | --------------------------------------------------------------- | ----------------------------------------------- |
    | 文档编号 | {前缀}-{序号}           | 前缀为S（阶段规范）/C（通用规范）/P（流程规范），序号为两位数字 | S01, C01, P01                                   |
    | 文档名称 | {阶段/领域名称}编写规范 | 简明扼要，体现文档的核心内容                                    | 战略规划编写规范                                |
    | 目的     | 一句话描述              | 说明本规范文档的编写目的                                        | 规定战略规划文档的章节结构、内容要求和编写模板  |
    | 范围     | 一句话描述              | 明确本规范文档的适用对象和管辖范围                              | 战略规划文档（文档编号01-战略规划）             |
    | 管理要点 | 关键词列表              | 列出本规范文档的核心管理要素，用顿号分隔                        | 正文9章结构、三层结构、四列章节结构表、表达规范 |
    | 文档密级 | 【密级标识】            | 标注文档的保密等级和处理要求                                    | 【商密二级】 -- 外发必追责                      |

    +   强制元数据清单

+   元数据的位置要求：
    -   所有强制元数据必须在第1章"总则"的开头，以列表形式呈现
    -   元数据顺序固定：文档编号 → 文档名称 → 目的 → 范围 → 管理要点 → 文档密级
    -   每项元数据占一行，格式为：`+   {元数据项}：{内容}`

+   章节特点：
    -   无下级章节，全部内容以列表形式呈现
    -   篇幅控制在0.5页以内
    -   不包含任何实质性规范内容

### 3.3.2 第2章：{规范对象}综述

+   章节定位：文档的导读章节，帮助读者快速理解规范对象的定位、章节结构和裁剪规则。
+   必须包含的内容：
    -   整体定位（包括定位图和定位说明）
    -   章节结构表
    -   裁剪规则
+   章节特点：
    -   固定为3个二级章节
    -   大量使用表格和图形，减少文字叙述
    -   篇幅控制在2-3页

#### 3.3.2.1 整体定位

+   内容要求：
    -   提供定位图，展示规范对象在研发过程中的位置、输入和输出
    -   配合定位说明，简要描述规范对象的作用和价值

+   定位图的标准格式：
    ```graph
    ┌─────────────────────────────────────────────────────────────┐
    │               {规范对象}在研发过程中的定位                  │
    ├─────────────────────────────────────────────────────────────┤
    │                                                             │
    │   输入                    阶段N                 输出        │
    │  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐   │
    │  │ 上游文档1    │───▶│              │───▶│ 本阶段文档   │   │
    │  │ 上游文档2    │    │  阶段名称    │    │              │   │
    │  │ ...          │    │              │    └──────┬───────┘   │
    │  └──────────────┘    └──────────────┘           │           │
    │                                                 ▼           │
    │                                          ┌──────────────┐   │
    │                                          │ 下游阶段     │   │
    │                                          └──────────────┘   │
    │                                                             │
    └─────────────────────────────────────────────────────────────┘
    ```

+   定位说明的内容要点：
    -   规范对象在研发流程中的位置
    -   主要输入来源（上游文档或信息）
    -   主要输出内容（本阶段文档）
    -   与下游阶段的关系

#### 3.3.2.2 章节结构表

+   内容要求：
    -   以图表方式约定规范对象应包含的章节
    -   明确各章节的必选/可选属性
    -   提供章节的简要说明

+   章节结构表的标准格式：

    | 章节编号 | 章节标题       | 必选/可选 | 章节说明                 |
    | -------- | -------------- | --------- | ------------------------ |
    | 第1章    | {章节标题}     | 必选      | {章节的作用和核心内容}   |
    | 第2章    | {章节标题}     | 必选      | {章节的作用和核心内容}   |
    | ...      | ...            | ...       | ...                      |

#### 3.3.2.3 裁剪规则

+   内容要求：
    -   说明哪些章节可以裁剪
    -   说明裁剪的条件和场景
    -   提供裁剪建议表

+   裁剪建议表的标准格式：

    | 章节编号 | 裁剪条件                 | 裁剪后影响               |
    | -------- | ------------------------ | ------------------------ |
    | 第X章    | {在什么情况下可以裁剪}   | {裁剪后对文档的影响}     |
    | 第Y章    | {在什么情况下可以裁剪}   | {裁剪后对文档的影响}     |

## 3.4 正文部分的结构要求

+   章节数量：根据规范对象的复杂度确定，通常为5-10章。
+   章节编号：从第3章开始，连续编号至第N章。
+   章节组织原则：
    -   按逻辑顺序组织：从宏观到微观、从整体到局部、从输入到输出
    -   每章聚焦一个核心主题，避免主题交叉
    -   章节之间应有清晰的逻辑递进关系

+   典型章节模式（参考阶段规范文档的共性结构）：
    -   模式1：按文档章节结构展开（适用于规定文档章节结构的规范）
        -   第3章：正文章节结构
        -   第4-N章：各章节的编写规范（每章一节）
    -   模式2：按内容维度展开（适用于规定文档内容要素的规范）
        -   第3章：内容框架
        -   第4-N章：各内容维度的规范（每维度一章）
    -   模式3：按流程阶段展开（适用于规定流程执行的规范）
        -   第3章：流程概览
        -   第4-N章：各阶段的执行规范（每阶段一章）

## 3.5 附录部分的结构要求

+   附录类型：
    -   规范性附录：包含强制性要求，与正文具有同等效力（标注"规范性"）
    -   资料性附录：提供参考信息，不具有强制性（标注"资料性"）

+   附录编号：
    -   使用大写字母编号：附录A、附录B、附录C...
    -   附录标题格式：`附录{字母}（{规范性/资料性}）{附录名称}`
    -   示例：`附录A（资料性）术语表`

+   附录排序：
    -   规范性附录（如果有）必须排在资料性附录之前
    -   同类附录内部按字母顺序排列
    -   修订记录不编号为附录，固定位于文档末尾

+   术语表的标准格式：
    -   分为两部分：业界通用术语（仅列名称）+ 文档专有术语（含定义）
    -   业界通用术语：AI已具备充分理解能力的术语，仅列出名称，不展开定义
    -   文档专有术语：本规范文档特有的术语或对通用术语的特定解释，需提供明确定义

+   修订记录的标准格式：
    -   位于文档末尾，不编号为附录
    -   标题固定为"修订记录"
    -   表格格式：

        | 标记          | 日期       | 修订内容       | 修订人   |
        | ------------- | ---------- | -------------- | -------- |
        | {版本号/R.xx} | YYYY-MM-DD | {修订内容概述} | {修订人} |

    -   修订记录按时间正序排列（最早的在上，最新的在下）

## 3.6 元数据的一致性要求

+   文档编号的一致性：
    -   文档编号应在文件名、元数据、文档标题中保持一致
    -   文件名格式：`{文档编号}-{文档名称}_v{版本号}@{时间戳}.md`（正式版本）或 `{文档编号}-{文档名称}_v{版本号}_R{修订号}@{时间戳}.md`（临时版本）
    -   示例：`S00-软件研发文档通用结构_v0.23@202602271350.md`

+   版本号的一致性：
    -   文件名中的版本号必须与文档内修订记录表中的最新版本号一致
    -   版本号递增规则遵循《版本管理规范》

+   管理要点的提取原则：
    -   管理要点应提取文档的核心结构特征和关键约束
    -   数量控制在3-6个关键词
    -   优先提取结构性要点（如"正文9章结构"），其次提取内容性要点（如"表达规范"）

# 4 章节组织规范（L2）

## 4.1 章节组织规范的作用

+   章节组织规范定义规范文档内部章节的层次结构和编排逻辑，确保章节之间的逻辑关系清晰、内容组织合理。
+   本章重点规范正文部分的章节组织方式，前置部分和附录部分的组织方式已在第3章明确。

## 4.2 三层结构模式

+   三层结构的定义：
    -   规范文档的正文章节应采用"章节综述 → 章节结构 → 表达规范"三层结构模式
    -   三层结构适用于规定文档章节编写规范的规范文档
    -   三层结构不适用于规定流程执行或技术标准的规范文档

+   三层结构的层次关系：

    ```text
    第N章 {章节名称}
    ├── N.1 章节综述
    │   └── 说明本章节的定位、作用、核心产出和向后传递关系
    ├── N.2 章节结构
    │   └── 定义本章节应包含的二级/三级章节及内容要点（四列表格）
    └── N.3 表达规范
        └── 定义本章节中各类内容的表达方式和格式要求（分项说明）
    ```

## 4.3 章节综述的编写要求

+   章节综述的作用：
    -   回答"本章节是什么、为什么需要、产出什么、如何传递"四个核心问题
    -   帮助读者快速理解章节的定位和价值，决定是否需要深入阅读

+   章节综述的必备要素：
    -   章节定位：本章节在文档中的作用和位置
    -   核心问题：本章节回答的核心业务问题（用"回答'...'"的句式）
    -   核心产出：本章节的主要交付物或分析成果
    -   向后传递：本章节的产出如何被后续章节或下游文档使用

+   章节综述的篇幅控制：
    -   通常为3-5个段落，每个段落1-3句话
    -   总篇幅控制在0.5页以内
    -   使用列表形式呈现，每个要素一个列表项

## 4.4 章节结构的编写要求

+   章节结构的作用：
    -   定义本章节应包含的二级/三级章节及其内容要点
    -   提供章节的"骨架"，指导文档编写者组织内容

+   四列章节结构表：
    -   标准格式：

        | 编号       | 标题       | 内容要点                             | 表达规范                              |
        | ---------- | ---------- | ------------------------------------ | ------------------------------------- |
        | {章节编号} | {章节标题} | {该章节应包含的核心内容，用分号分隔} | {该章节适用的表达规范名称，引用N.3节} |

    -   表格说明：
        -   编号列：填写二级或三级章节编号（如1.1、1.2.1）
        -   标题列：填写章节标题，简明扼要
        -   内容要点列：列出该章节应包含的核心内容，多个要点用分号分隔
        -   表达规范列：引用本章N.3节定义的表达规范名称，无特定规范时留空

+   章节结构表的编写原则：
    -   覆盖本章节的所有二级章节，三级章节可选择性列出
    -   内容要点应具体明确，避免笼统描述
    -   表达规范应与N.3节的定义一一对应

## 4.5 表达规范的编写要求

+   表达规范的作用：
    -   定义本章节中各类内容的表达方式和格式要求
    -   提供可复用的内容模板，降低文档编写的不确定性

+   表达规范的组织方式：
    -   按内容类型分项说明，每种内容类型一个三级章节
    -   三级章节标题格式：`N.3.{序号} {内容类型名称}`
    -   示例：`3.3.1 领域术语表`、`3.3.2 术语映射表`

+   表达规范的必备要素：
    -   表格类内容：提供列定义表 + 填写示例
    -   图形类内容：提供绘制约定 + 示例图
    -   文本类内容：提供格式模板 + 填写说明

+   列定义表的标准格式：

    | 列名    | 填写说明                   |
    | ------- | -------------------------- |
    | {列名1} | {该列的填写要求和取值说明} |
    | {列名2} | {该列的填写要求和取值说明} |
    | ...     | ...                        |

    +   {内容类型名称}列定义

+   填写示例的提供原则：
    -   每种表达规范至少提供一个完整的填写示例
    -   示例应具有代表性，覆盖典型场景
    -   示例应与列定义表保持一致

## 4.6 章节层级深度控制

+   层级深度限制：
    -   规范文档的章节层级深度不应超过四级（章-节-条-款）
    -   标准层级：一级标题（章）→ 二级标题（节）→ 三级标题（条）→ 四级标题（款）
    -   超过四级时应考虑章节拆分或内容重组

+   层级编号规则：
    -   一级标题：阿拉伯数字（1, 2, 3...）
    -   二级标题：一级编号.阿拉伯数字（1.1, 1.2, 1.3...）
    -   三级标题：二级编号.阿拉伯数字（1.1.1, 1.1.2...）
    -   四级标题：三级编号.阿拉伯数字（1.1.1.1, 1.1.1.2...）

+   层级使用建议：
    -   一级标题：文档的主要章节，数量控制在10章以内
    -   二级标题：章节的主要分节，每章通常3-6节
    -   三级标题：节的细分条目，每节通常2-5条
    -   四级标题：谨慎使用，仅在内容确实需要进一步细分时使用

# 5 内容要素框架（L3）

## 5.1 内容要素框架的作用

+   内容要素框架定义规范文档中各类章节应包含的核心要素类型，确保内容的完整性和一致性。
+   内容要素分为三类：
    -   强制要素：所有同类章节必须包含的要素，缺失将导致内容不完整
    -   推荐要素：建议包含的要素，有助于提高内容质量
    -   可选要素：根据具体情况选择性包含的要素

## 5.2 要素分类的指导原则

+   规范文档应根据章节的功能定位，明确该章节应包含的要素类型。
+   常见的章节功能类型包括：
    -   定义类：定义概念、术语、对象、规则
    -   流程类：描述执行步骤、操作流程、方法论
    -   规范类：规定编写规则、格式要求、质量标准
    -   分析类：进行问题分析、需求分析、架构分析
    -   映射类：建立不同层次、不同文档之间的对应关系
    -   裁剪类：说明规范文档或交付文档的裁剪规则

+   各规范文档应在自身的章节结构定义中，明确每个章节的功能类型和必备要素。

## 5.3 要素完整性检查

+   规范文档应提供要素完整性检查清单，确保各章节包含必要的要素。
+   检查清单应包括：
    -   章节编号和标题
    -   章节功能类型
    -   强制要素清单
    -   推荐要素清单
    -   实际包含的要素

+   要素完整性检查应在文档编写阶段和评审阶段分别执行。

    -   列定义表：定义表格列结构的标准表格，包含列名和填写说明两列
    -   可判断标准：定义的质量标准，要求读者能根据定义判断某个实例是否属于该定义对象
