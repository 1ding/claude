# 1 目的和范围

+   文档编号：S10
+   文档名称：软件工程文档编写规范的编写规范
+   目的：规定软件工程领域"技术文档编写规范"类文档的专用体例
+   范围：所有软件工程领域的"技术文档编写规范"类文档（以下简称"编写规范"），即以"章节结构表 + 表达规范"为核心手法，约束某类技术文档怎么写的规范文档。示例：《S01-战略规划编写规范》、宏观需求规格说明书编写规范、架构设计文档编写规范等
+   管理要点：三层结构、四列章节结构表、表达规范
+   文档密级：【商密二级】 -- 外发必追责。

# 2 编写规范结构要求

## 2.1 编写规范的完整结构

+   软件工程文档编写规范的分区结构遵循《C11-文档通用结构规范》定义的五分区结构（总则、正文、规范性附录、资料性附录、修订记录）。
+   各分区的编写规则来源如下表所示。

    | 分区       | 规则来源                                                                        |
    | ---------- | ------------------------------------------------------------------------------- |
    | 总则       | 《C11-文档通用结构规范》第5节 + 《M01-规范文档编写规范》第2.2节 + 本规范第2.2节 |
    | 正文       | 《M01-规范文档编写规范》第2.3节 + 本规范第2.3节                                 |
    | 规范性附录 | 《C11-文档通用结构规范》第6.1节                                                 |
    | 资料性附录 | 《C11-文档通用结构规范》第6.1节 + 《M01-规范文档编写规范》第2.4节               |
    | 修订记录   | 《C11-文档通用结构规范》第7节                                                   |

## 2.2 总则要求

+   软件工程文档编写规范的总则按《C11-文档通用结构规范》第5节和《M01-规范文档编写规范》第2.2节执行。
+   无扩展要求。

## 2.3 正文要求

+   软件工程文档编写规范的正文称为"规则主体"，按《M01-规范文档编写规范》第2.3节执行。
+   在M01基础上，软件工程文档编写规范的规则主体采用"三层结构"组织——为目标文档的每个章节（含正文章节和附录）分别定义编写规则。

### 2.3.1 三层结构

+   每个目标文档章节的定义由以下三层组成，顺序固定。

    | 层次   | 子标题命名     | 必备/可选 | 用途                               |
    | ------ | -------------- | --------- | ---------------------------------- |
    | 第一层 | X.Y.1 章节综述 | 必备      | 说明章节的功能定位、读者和信息流向 |
    | 第二层 | X.Y.2 章节结构 | 必备      | 定义章节要写什么内容、有哪些约束   |
    | 第三层 | X.Y.3 表达规范 | 可选      | 定义具体内容的语义约束和内容要求   |

+   标题编号规则："X.Y"中 X 为编写规范的章节号，Y 为目标文档的章节序号。若目标文档章节为多级编号（如3.2.1），Y 取完整序号（如"2.3.2.1 章节综述"）。示例：S01号规范中"2.1 第1章 执行摘要"→"2.1.1 章节综述""2.1.2 章节结构""2.1.3 表达规范"。

### 2.3.2 第一层：章节综述

+   章节综述说明该章节的功能定位、主要读者和信息流向，应回答以下三个问题：

    | 问题             | 说明                                       |
    | ---------------- | ------------------------------------------ |
    | 本章解决什么问题 | 该章节在目标文档中的功能定位               |
    | 面向谁           | 该章节的主要读者角色                       |
    | 向后传递什么     | 本章结论为后续哪些章节提供输入（信息流向） |

### 2.3.3 第二层：章节结构

+   章节结构以四列表定义目标文档某章节的子章节结构、内容要点和表达规范索引。四列分别为：编号、标题、内容要点、表达规范。
+   内容要点应是可执行的编写指令。判断标准：编写者读后能明确知道要写什么、写到什么粒度。
+   内容要点应标注强制性级别：(必选)、(宜)、(可选)。未标注视同 (必选)。
+   子章节数量不固定时，编号用 `x` 占位（如"3.2.x"），标题用变量占位（如"{客户类型}"）。固定居末的汇总子章节，编号用 `n` 占位（如"2.1.n"），标题为汇总性标题（如"共同根源"）。
+   以下为S01号规范第2章的章节结构示例：

    | 编号  | 标题         | 内容要点                                         | 表达规范               |
    | ----- | ------------ | ------------------------------------------------ | ---------------------- |
    | 2.1   | 核心痛点分析 | 按"困境还原→根因分析"逐个展开                    |                        |
    | 2.1.x | 困境x        | 场景还原（流程文本或伪代码）                     | 2.1.3 困境还原规范     |
    |       |              | 困境表现（维度、现状、影响范围、严重程度）       | 2.1.3 困境表现规范     |
    |       |              | 根本原因                                         |                        |
    | 2.1.n | 共同根源     | 多困境归纳为共同根源                             |                        |
    |       |              | 困境→根源→解决方案关系图                         | 2.1.3 困境归因规范     |
    | 2.2   | 目标市场定义 |                                                  |                        |
    | 2.2.1 | 市场选择逻辑 | 首选场景选择依据                                 | 2.2.3 场景选择依据规范 |
    | 2.2.2 | 客户分层     | 分层（特征、产品形态、客单价、目标）             | 2.2.3 客户分层规范     |
    | 2.2.3 | 市场容量估算 | 分客户类型估算潜在规模，标注估算依据（数据来源） | 2.2.3 市场容量估算规范 |
    |       |              | 目标渗透率 + 行业基准对比论证合理性              |                        |
    | 2.3   | 市场窗口期   | 窗口形成条件 + 时间判断 + 先发优势               |                        |
    | 2.4   | 增长驱动因素 | 驱动市场增长的核心因素                           |                        |

### 2.3.4 第三层：表达规范

+   表达规范为目标文档的具体内容定义语义约束和内容要求。规范形式不限——可以是列说明表、流程约束、文字描述，或其组合。
+   每个表达规范以子标题独立命名，格式为"{内容名称}规范"，如"困境还原规范""客户分层规范""部署拓扑规范"。
+   规范定义在首次被引用的章节的 X.Y.3 小节，后续章节以"见 X.Y.3 规范名"引用。
+   每个规范条目由以下部分组成，顺序固定：

    | 部分     | 必备/可选 | 说明                                                       |
    | -------- | --------- | ---------------------------------------------------------- |
    | 规范     | 必备      | 定义各字段/元素的语义、约束条件、内容要求                  |
    | 补充说明 | 可选      | 规范本身不够自明时，补充填写指引、判断标准、注意事项       |
    | 示例     | 可选      | 规范较复杂或易产生歧义时，给出一个填写后的示例帮助消除歧义 |

+   表达规范示例——以第2.3.3节章节结构示例中"2.1.3"小节为例：

    ```text
        ### 2.1.3 表达规范

        #### 困境还原规范

        | 元素     | 说明                                                         |
        | -------- | ------------------------------------------------------------ |
        | 步骤编号 | 从1开始连续递增，每个困境独立编号                            |
        | 角色     | 执行操作的用户角色，如"需求分析师""项目经理"                 |
        | 动作     | 具体操作，用动宾结构描述，如"编写需求文档""评审设计方案"     |
        | 系统响应 | 系统或环境对该动作的反馈，如"生成报告""触发审批流程"         |
        | 断点标注 | 用"⚠ 断点："标注暴露核心问题的步骤，每个困境有且仅有一个断点 |

        补充说明：以流程文本或伪代码还原用户在困境中的实际操作步骤。
        每个困境独立一节，步骤数量不限，断点标注在暴露核心问题的步骤之后。

        示例：
        ```
        步骤 1. 需求分析师 编写需求文档 → 生成Word文档
        步骤 2. 需求分析师 发送邮件给评审组 → 收到自动回复
        步骤 3. 评审组成员 在各自时间阅读文档 → 无统一反馈渠道
        ⚠ 断点：评审意见分散在邮件、即时消息、会议记录中，无法追溯
        ```

        #### 困境表现规范

        | 列名     | 说明                                                     |
        | -------- | -------------------------------------------------------- |
        | 表现维度 | 困境在哪个层面可观测，如效率、质量、成本、体验           |
        | 现状描述 | 当前实际表现，用可量化指标或可观察行为描述               |
        | 影响范围 | 受影响的角色、流程或系统，如"全部一线销售""订单处理流程" |
        | 严重程度 | 高/中/低，附判断依据                                     |

        补充说明：现状描述应优先使用可量化指标（如"平均耗时3小时""错误率15%"），
        无法量化时使用可观察行为（如"需要人工逐个检查""依赖口头传达")。
    ```

+   在C11和M01基础上，软件工程文档编写规范的每个附录（无论规范性或资料性）均采用与正文章节相同的三层结构：X.Y.1 章节综述、X.Y.2 章节结构（四列表）、X.Y.3 表达规范（如有）。编写规则按第2.3节执行。

# 附录A（资料性）术语表

+   业界通用术语（AI已具备充分理解能力，仅列出名称）：
    -   ADR、ASCII图、UML、DAG

+   天帷通用术语（定义参见《C99-天帷通用术语规范》）：
    -   语义ID、短号、类型前缀

+   文档专有术语：
    -   软件工程文档编写规范：以"章节结构表 + 表达规范"为核心手法，约束某类软件工程技术文档各章节的编写规则、内容要求和判断标准的规范文档。
    -   章节结构表：以四列（编号、标题、内容要点、表达规范）定义目标文档章节结构的表格，是软件工程文档编写规范的核心编写手法。
    -   三层结构：章节定义的固定组织方式，由章节综述（X.Y.1）、章节结构（X.Y.2）、表达规范（X.Y.3）三层按固定顺序组成。
    -   表达规范：为目标文档某类内容定义语义约束和内容要求的规范，形式不限（列说明表、流程约束、文字描述，或其组合），定义在首次被引用章节的 X.Y.3 小节。每个规范条目由规范（必备）、补充说明（可选）、示例（可选）三部分组成。
    -   四列约定：章节结构表的列结构约定——编号、标题、内容要点、表达规范。
    -   归属规则：表达规范定义位置的确定规则——规范定义在首次被引用的章节，后续章节以"见"前缀引用。
    -   强制性标注：内容要点的强制性级别标注体系，分为 (必选)、(宜)、(可选) 三级，用于标注章节结构表中各内容要点的强制程度，为裁剪提供依据。
