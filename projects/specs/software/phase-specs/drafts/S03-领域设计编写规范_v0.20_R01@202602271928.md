# 2 领域设计文档结构要求

+   领域设计文档是阶段3"领域设计"的唯一正式输出物，为后续阶段4"系统设计"提供输入。文档通过领域设计评审后，阶段3结束。

+   领域设计文档的标准章节结构如下。

    | 章节  | 标题                         | 必备/可选 | 说明                                                                           |
    | ----- | ---------------------------- | --------- | ------------------------------------------------------------------------------ |
    | 第1章 | 文档目的与方法论框架         | 必备      | 导读章节：文档目的、领域设计方法论框架图和术语约定，不含领域分析内容           |
    | 第2章 | 业务知识获取                 | 必备      | 将阶段2三线分析成果和领域专家知识转化为领域设计输入，产出问题空间全景          |
    | 第3章 | 统一语言                     | 必备      | 建立领域术语表（含定义、子域/BC归属、阶段2术语映射）和术语治理规则             |
    | 第4章 | 领域划分与子域分类           | 必备      | 识别子域并按核心域/支撑域/通用域分类，产出子域清单和 Business Epic 覆盖关系    |
    | 第5章 | 限界上下文识别与定义         | 必备      | 划定BC边界，产出BC清单（含职责、核心聚合、拥有的实体），验证BC内聚性           |
    | 第6章 | 上下文映射                   | 必备      | 定义BC之间的协作关系（映射模式、集成策略、数据流向），产出上下文映射图         |
    | 第7章 | 领域事件全景                 | 必备      | 识别领域事件（域内/跨域），产出事件清单和事件流转全景图                        |
    | 第8章 | 核心领域模型概要             | 必备      | 对核心域BC进行聚合级初步建模（聚合根、核心实体、关键值对象），不展开属性和方法 |
    | 第9章 | 架构需求分析（Enabler Epic） | 必备      | 将跨BC架构需求显性化为 Enabler Epic，产出架构约束传递清单                      |

+   当某项内容在特定项目中不适用时，允许裁剪。(必选)项不可裁剪，如确实不适用应保留章节标题并注明"本项目不适用，原因：{具体原因}"。(宜)或(可选)项可直接省略。裁剪后仍应确保追溯链完整性（Business Epic → 子域 → BC → Enabler Epic 的链路不断裂）。

+   裁剪建议：

    | 项目规模 | 判断参考               | 裁剪策略                                                                                          |
    | -------- | ---------------------- | ------------------------------------------------------------------------------------------------- |

    | 小型项目 | BC < 3，跨域交互 <= 1  | 第2章可简化（省略EventStorming等重方法）；第7章可合并入第6章；第8章可省略（留至阶段6 BC总体设计） |

## 3.3 第3章 统一语言

## 3.4 第4章 领域划分与子域分类

### 3.4.1 章节综述

+   本章完成从问题空间到解决方案空间的第一步战略决策，回答"业务应如何分解为子域，各子域的战略地位如何"——从第2章的问题空间全景出发，识别子域并按核心域、支撑域、通用域分类，子域分类直接影响投资策略。
+   核心产出为子域清单（含分类、职责定义、投资策略）、子域与业务域映射关系、子域与 Business Epic 覆盖关系。
+   本章子域清单向后传递至第5章，作为BC识别的问题空间框架；子域分类向后传递至第8章，作为建模深度决策的依据。
