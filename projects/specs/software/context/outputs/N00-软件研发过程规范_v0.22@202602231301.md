# 1 适用范围

+   文档编号：N00
+   文档名称：软件研发过程规范
+   适用对象：公司所有软件产品的研发项目
+   管辖范围：软件研发过程的阶段划分、各阶段的目标、关键活动、输入输出及交付物要求、过程管理要求（阶段门禁、变更管理、可追溯性、质量保障）
+   不管辖的范围：各类交付物的具体编写规范（见各交付物专用编写规范）、编号体系（见《P01-软件开发编号规范》）、术语定义（见《C99-天帷通用术语规范》）、错误代码（见《P02-软件开发错误代码规范》）
+   本规范与其他规范的关系：本规范定义研发过程框架，各阶段交付物的编写标准由独立的编写规范约束；编号规范和错误代码规范中的阶段划分应与本规范保持一致
+   文档密级：【商密二级】 -- 外发必追责。

# 2 研发过程总览

+   软件研发过程分为8个阶段，各阶段从宏观到微观、从业务到技术逐步细化。

    ```graph
    阶段1 产品战略 → 阶段2 宏观需求 → 阶段3 领域设计 → 阶段4 系统设计 → 阶段5 BC详细需求 → 阶段6 BC总体设计 → 阶段7 BC详细设计 → 阶段8 开发实现
    ```

+   各阶段之间存在顺序依赖关系，实际执行中可根据项目特点迭代和并行。
+   8个阶段与三个需求域的对应关系如下（详见附录B）：

    ```graph
    需求域              对应阶段                        SAFe 需求层级
    ─────────────────────────────────────────────────────────────────────────
    业务需求域          阶段1 产品战略                  Strategic Theme
                        阶段2 宏观需求                  Business Epic
    ─────────────────────────────────────────────────────────────────────────
    系统架构需求域      阶段3 领域设计                  Enabler Epic
                        阶段4 系统设计                  Capability / Enabler Capability / Enabler Feature
    ─────────────────────────────────────────────────────────────────────────
    软件开发需求域      阶段5 BC详细需求                Feature → User Story（由Capability拆分而来）
                        阶段6 BC总体设计
                        阶段7 BC详细设计                Task
                        阶段8 开发实现
    ─────────────────────────────────────────────────────────────────────────
    ```

# 3 阶段1：产品战略

## 3.1 阶段目标

+   明确产品的战略定位、目标市场、核心价值主张和商业模式，定义 Strategic Theme（战略主题），为后续研发提供方向性指引。

## 3.2 关键活动

-   市场分析与竞品研究
-   产品愿景与定位确定
-   Strategic Theme 定义：明确投资组合级的战略方向和业务目标
-   商业模式设计与可行性评估
-   产品路线图规划（基于 Strategic Theme 分解中长期演进计划）
-   关键成功指标（KPI）定义

## 3.3 阶段输入

-   市场调研报告
-   公司战略规划
-   客户反馈与需求洞察
-   技术趋势分析

## 3.4 阶段输出

| 交付物       | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| 产品战略文档 | 含产品愿景、目标市场、价值主张、商业模式、竞争策略、         |
|              | Strategic Theme 清单等，具体格式参见产品战略文档编写规范     |
| 产品路线图   | 分阶段的产品演进计划，标明关键里程碑，各里程碑可追溯到对应的 |
|              | Strategic Theme                                              |

# 4 阶段2：宏观需求

## 4.1 阶段目标

+   从整体业务视角梳理和定义产品的宏观需求，将 Strategic Theme 分解为 Business Epic（业务史诗），识别核心业务流程和关键用户场景，建立需求的全局视图。

## 4.2 关键活动

-   业务流程梳理与建模
-   系统角色与场景分析
-   Business Epic 识别与定义：将 Strategic Theme 分解为端到端的业务目标或价值流
-   核心功能需求识别与分类
-   非功能需求（性能、安全、可用性等）初步定义
-   Business Epic 优先级排序

## 4.3 阶段输入

-   产品战略文档（含 Strategic Theme 清单）
-   产品路线图
-   利益相关方访谈记录
-   现有系统分析（如适用）

## 4.4 阶段输出

| 交付物       | 说明                                                                 |
| ------------ | -------------------------------------------------------------------- |
| 宏观需求文档 | 涵盖业务全景、核心流程、用户场景、Business Epic 清单（含优先级与验收 |
|              | 标准）及非功能需求，具体格式参见宏观需求文档编写规范                 |
| 业务流程图   | 核心业务流程的可视化表达                                             |

# 5 阶段3：领域设计

## 5.1 阶段目标

+   运用领域驱动设计方法，对业务领域进行合理划分，识别核心领域、支撑领域和通用领域，建立统一语言。同时识别和定义 Enabler Epic（架构史诗），将跨BC的架构决策显性化管理。

## 5.2 关键活动

-   领域知识获取与统一语言建立
-   领域划分（核心域、支撑域、通用域）
-   限界上下文（BC）识别与边界划定
-   BC间关系与上下文映射（Context Map）设计
-   领域事件识别
-   Enabler Epic 识别与定义：基于 Business Epic 和领域划分结果，识别跨BC的架构决策需求（如领域边界划定的技术约束、BC间通信模式选型、共享内核策略等）

## 5.3 阶段输入

-   宏观需求文档（含 Business Epic 清单）
-   业务流程图
-   领域专家知识

## 5.4 阶段输出

| 交付物       | 说明                                                                   |
| ------------ | ---------------------------------------------------------------------- |
| 领域设计文档 | 包括领域划分、BC识别、上下文映射图、统一语言词汇表等，具体格式参见领域 |
|              | 设计文档编写规范。文档中应包含"架构需求分析"章节，记录本阶段识别的     |
|              | Enabler Epic 清单及其与 Business Epic 的追溯关系                       |

# 6 阶段4：系统设计

## 6.1 阶段目标

+   基于领域设计成果，完成系统层面的整体架构设计，确定技术选型、部署架构、集成策略和关键技术方案。将 Enabler Epic 分解为 Enabler Capability（架构能力）和 Enabler Feature（架构特性），形成可落地的架构需求清单。

## 6.2 关键活动

-   系统整体架构设计
-   技术选型与评估
-   BC到服务/模块的映射
-   系统集成方案设计（API网关、消息机制等）
-   基础设施与部署架构设计
-   系统级非功能需求方案设计（安全、性能、可观测性等）
-   Enabler Capability 定义：将 Enabler Epic 分解为跨BC的架构级能力（如统一认证体系、跨服务事件总线、可观测性平台等）
-   Enabler Feature 定义：将 Enabler Capability 进一步分解为具体的架构特性（如具体的技术选型方案、集成接口规范、非功能需求指标要求等），明确验收标准

## 6.3 阶段输入

-   领域设计文档（含 Enabler Epic 清单）
-   宏观需求文档（非功能需求部分）
-   技术约束与现有架构资产

## 6.4 阶段输出

| 交付物       | 说明                                                                     |
| ------------ | ------------------------------------------------------------------------ |
| 系统设计文档 | 包括系统架构图、技术选型说明、集成方案、部署架构、非功能需求方案等，具体 |
|              | 格式参见系统设计文档编写规范。文档中应包含"架构需求清单"章节，记录       |
|              | Enabler Capability 和 Enabler Feature 列表（含验收标准），并标注与上游   |
|              | Enabler Epic 的追溯关系                                                  |

# 7 阶段5：BC详细需求

## 7.1 阶段目标

+   在宏观需求和领域设计的基础上，将 Business Epic 和 Capability 逐步拆分为 Feature（特性）和 User Story（用户故事），针对每个BC进行细粒度的需求分析和定义，产出可指导设计的详细需求规格。

## 7.2 关键活动

-   Capability（含 Enabler Capability）拆分：将跨BC的大粒度能力拆分为各BC可独立交付的 Feature
-   Feature 定义：明确每个 Feature 的业务价值假设和验收标准
-   User Story 编写：将 Feature 细化为单个可验收的用户故事
-   需求验收标准定义：为每个 Feature 和 User Story 定义明确的、可测试的验收标准
-   业务规则详细定义
-   数据需求分析
-   BC级别的非功能需求细化（承接系统设计阶段的 Enabler Feature 约束）
-   BC间交互需求明确

## 7.3 阶段输入

-   宏观需求文档（含 Business Epic 清单）
-   领域设计文档（含 Enabler Epic 清单）
-   系统设计文档（含 Enabler Capability / Enabler Feature 架构需求清单）

## 7.4 阶段输出

| 交付物         | 说明                                                                     |
| -------------- | ------------------------------------------------------------------------ |
| BC详细需求文档 | 每个BC对应一份，包含 Feature 列表、User Story（含验收标准）、业务规则、  |
|                | 数据需求等，具体格式参见BC详细需求文档编写规范。各 Feature 和 User Story |
|                | 应可追溯到上游 Business Epic，并标注受上游 Enabler Feature 约束的        |
|                | 非功能需求                                                               |

# 8 阶段6：BC总体设计

## 8.1 阶段目标

+   针对每个BC，完成总体级别的技术设计。所有BC均需确定BC内部的模块划分、核心领域模型、关键接口和数据存储方案，并将 User Story 分解为初步的 Task（任务），建立需求到设计的追溯关系。
+   大BC：在总体设计共性内容之外，还需完成子领域划分、跨子领域约束定义、公共架构决策。
+   小BC：按传统方式完成总体设计，无需子领域划分。

## 8.2 BC规模判定

+   BC是否需要划分子领域，应在本阶段启动时判定。满足以下任一条件的BC判定为大BC：

-   BC内包含多个可独立演进的业务关注点
-   BC内的 Feature 数量或复杂度导致单份详细设计文档难以维护
-   BC内存在可由不同团队或人员并行设计的独立模块

+   不满足以上任一条件的BC按小BC处理。

## 8.3 关键活动

+   所有BC的共性活动：

-   BC内部模块/层次结构设计
-   聚合根与领域模型总体设计
-   对外接口（API）概要定义
-   数据存储方案设计
-   BC内关键技术方案设计
-   与其他BC的集成接口设计（落实系统设计阶段的 Enabler Feature 集成约束）
-   Task 初步分解：基于 User Story 和总体设计方案，识别主要的实现任务

+   大BC的额外活动：

-   子领域划分：将BC内部按业务关注点划分为若干子领域，明确每个子领域的职责边界
-   跨子领域约束定义：明确子领域之间的依赖关系、数据共享规则、接口契约
-   公共架构决策：确定子领域共享的基础设施选型、公共模块、统一的领域事件规范等

## 8.4 阶段输入

-   BC详细需求文档（含 Feature / User Story 清单）
-   系统设计文档（含架构需求清单中的 Enabler Feature 约束）
-   领域设计文档

## 8.5 阶段输出

| 交付物         | 说明                                                                   |
| -------------- | ---------------------------------------------------------------------- |
| BC总体设计文档 | 每个BC对应一份。小BC：包含模块结构、领域模型概要、接口概要、数据方案、 |
|                | 初步 Task 清单等。大BC：在上述内容基础上，还应包含子领域划分方案、跨子 |
|                | 领域约束、公共架构决策，以及各子领域的概要级设计。设计方案应可追溯到   |
|                | Feature / User Story，并标注对 Enabler Feature 约束的落实方式          |

# 9 阶段7：BC详细设计

## 9.1 阶段目标

+   在总体设计基础上，完成BC的详细技术设计，产出可直接指导编码实现的设计方案。细化 Task 至可分配给开发人员的粒度。
+   大BC：按子领域分别产出详细设计文档，每个子领域一份（共 N 份）。
+   小BC：整个BC产出一份详细设计文档。

## 9.2 关键活动

-   详细领域模型设计（实体、值对象、聚合、领域服务）
-   API详细设计（请求/响应结构、错误码、版本策略）
-   数据模型详细设计（表结构、索引、数据迁移）
-   核心业务逻辑与算法设计
-   异常处理与边界条件设计
-   单元测试策略设计
-   Task 细化与工作量估算：将初步 Task 细化为可在迭代内完成的具体实现任务

+   大BC的额外要求：

-   各子领域的详细设计应遵循总体设计中定义的跨子领域约束和公共架构决策
-   子领域之间的接口契约应在各自详细设计文档中同步定义，保持一致

## 9.3 阶段输入

-   BC总体设计文档（含初步 Task 清单；大BC还含子领域划分方案和跨子领域约束）
-   BC详细需求文档（含 User Story 及验收标准）

## 9.4 阶段输出

| 交付物         | 说明                                                                   |
| -------------- | ---------------------------------------------------------------------- |
| BC详细设计文档 | 小BC：每个BC对应一份，包含详细领域模型、API规格、数据模型、算法设计、  |
|                | 细化后的 Task 清单（含工作量估算）等。大BC：每个子领域对应一份，内容   |
|                | 同上，各子领域文档应可追溯到总体设计中的子领域划分方案，并遵循跨子领域 |
|                | 约束。具体格式参见BC详细设计文档编写规范                               |

# 10 阶段8：开发实现

## 10.1 阶段目标

+   依据详细设计文档，按 Task 分配开发工作，完成编码实现、单元测试、集成测试和系统测试，交付可部署的软件产品。

## 10.2 关键活动

-   按 Task 分配开发工作，在迭代内完成编码实现
-   代码审查
-   单元测试与集成测试（验证 User Story 验收标准和 Enabler Feature 非功能需求）
-   持续集成/持续部署（CI/CD）
-   系统测试与验收测试
-   缺陷修复与回归测试
-   部署与发布

## 10.3 阶段输入

-   BC详细设计文档（含细化 Task 清单）
-   BC总体设计文档
-   BC详细需求文档（含 User Story 验收标准）
-   系统设计文档（含 Enabler Feature 非功能需求验收标准）

## 10.4 阶段输出

| 交付物            | 说明                                                    |
| ----------------- | ------------------------------------------------------- |
| 源代码            | 经代码审查通过，符合编码规范，可追溯到对应 Task 和      |
|                   | User Story                                              |
| 测试报告          | 包括单元测试、集成测试、系统测试的结果，覆盖 User Story |
|                   | 验收标准和 Enabler Feature 非功能需求                   |
| 部署包与部署文档  | 可部署的软件制品及部署说明                              |
| 用户手册/操作指南 | 如适用                                                  |

# 11 过程管理要求

## 11.1 阶段门禁

+   每个阶段完成后，应进行阶段评审，确认交付物质量和完整性，方可进入下一阶段。评审可根据项目规模和风险等级采取不同形式（正式评审会、同行评审、检查单确认等）。

## 11.2 变更管理

+   各阶段产出物如需变更，应遵循变更管理流程，评估变更影响范围并获得相关方确认。前序阶段的变更需评估对后续阶段的影响并同步更新。

## 11.3 可追溯性

+   各阶段交付物之间应建立可追溯的关联关系，确保从产品战略到代码实现的双向追溯能力。
+   推荐的追溯链路（括号内为对应的SAFe需求层级）：

    ```graph
    Strategic Theme → Business Epic → Enabler Epic → Enabler Capability    → Feature      → User Story → Task
    （产品战略）      （宏观需求）     （领域设计）    / Enabler Feature       （BC详细需求）  （BC详细需求） （设计与实现）
                                                    （系统设计）
                                                         ↓
                                                     Capability
                                                    （跨BC能力）
                                                         ↓
                                                      Feature
                                                    （BC详细需求）
    ```

+   大BC的追溯应额外覆盖：BC总体设计 → 子领域划分 → 子领域详细设计文档，确保每份子领域详细设计文档可追溯到总体设计中的子领域定义和跨子领域约束。
+   追溯应覆盖：需求项 ↔ 设计文档 ↔ 源代码 ↔ 测试用例，实现全链路的正向追踪与反向溯源。

## 11.4 质量保障

+   各阶段应遵循相应的编写规范和质量标准。各类交付物的具体编写规范由独立的规范文档定义，本文档不重复约定。

+   BABOK v3 将需求分为 Business、Stakeholder、Solution（含 Functional / Non-functional）、Transition 四类。Stakeholder Requirements 是 Business Requirements 的角色视角补充，Transition Requirements 是实施过渡期的临时性需求，完成后即失效。
+   ISO/IEC/IEEE 15288/12207/29148 是一组配套标准。15288（系统生命周期过程）定义了 Stakeholder Requirements 和 System Requirements 两层需求过程；12207（软件生命周期过程）与 15288 共享相同的过程模型，增加了 Software Requirements 层；29148（需求工程）为这三层过程分别定义了规格文档：StRS → SyRS → SRS，从系统整体逐步细化到软件实现层面。
+   敏捷体系按交付粒度分层：Theme → Epic → Feature → User Story → Task。
+   SAFe 在敏捷基础上增加了 Capability（能力）层级用于跨ART协调，并为每个层级引入 Enabler 变体（Enabler Epic、Enabler Capability、Enabler Feature、Enabler Story），使架构、基础设施、合规等技术工作与业务需求项遵循相同的管理规则。
+   四套体系从不同视角对需求进行分层，实际项目中可根据团队实践选用或组合使用。

# 附录B（资料性）推荐需求模型

+   本规范推荐直接采用 SAFe 需求层级体系管理各阶段的需求。SAFe 体系天然覆盖业务需求、系统架构需求和软件开发需求三个层面——业务价值通过 Business Epic / Capability / Feature 承载，架构与基础设施工作通过对应的 Enabler 变体显性化管理，两者遵循相同的优先级排序和验收规则。

+   三个需求域与 ISO/IEC/IEEE 15288/12207/29148 标准族的三层需求框架（StRS / SyRS / SRS）自然对应，明确了需求从业务到系统到软件的层次递进关系。
+   Capability（能力）是 SAFe 中介于 Epic 和 Feature 之间的层级，用于描述跨BC（跨ART）的大解决方案能力，需要多个BC协作交付。Capability 拆分为多个 Feature 后分配到各BC实现。Enabler Capability 承载跨BC的架构级能力（如统一认证体系、跨服务事件总线等）。
+   Enabler 是所有项目的固有组成部分，无论项目规模大小，架构决策、基础设施搭建、技术选型等工作客观存在。大型项目应显式管理 Enabler Epic、Enabler Capability 和 Enabler Feature，确保架构工作可见、可排期、可验收；小型项目可简化管理（如省略 Capability 层级，或将 Enabler 合并到相关 Business 项中跟踪），但不应忽视其存在。
+   Business 和 Enabler 遵循相同规则：均需明确验收标准，均参与优先级排序，均可增量交付和度量。二者的区别仅在于价值导向不同——Business 直接交付用户价值，Enabler 构建架构跑道以支撑未来的业务交付。
+   各需求域之间应保持纵向一致性：上层需求是下层需求的来源和依据，下层需求是上层需求的细化和落地。跨域的需求变更需评估对下游所有层级的影响。

# 附录C（资料性）追溯链全景

+   各阶段产出物之间应建立如下追溯关系：

    ```graph
    产品战略文档
      └── 宏观需求文档
            ├── 业务流程图
            └── 领域设计文档
                  ├── 上下文映射图
                  └── 系统设计文档
                        └── BC详细需求文档 (per BC)
                              └── BC总体设计文档 (per BC)
                                    ├── [小BC] BC详细设计文档 (1份/BC)
                                    │         ├── 源代码
                                    │         └── 测试用例
                                    └── [大BC] 子领域详细设计文档 (N份/BC)
                                              ├── 源代码
                                              └── 测试用例
    ```

+   正向追溯：从产品战略出发，可追踪到每一行代码和测试用例所支撑的业务目标。
+   反向追溯：从任意代码或缺陷出发，可回溯到其对应的设计决策、需求来源和战略依据。
+   变更影响分析：当某一层级的产出物发生变更时，可快速识别受影响的下游产出物。
+   宜在项目管理工具中为各层级产出物建立关联关系，实现自动化的追溯与变更影响分析。
