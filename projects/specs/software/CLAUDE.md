# 软件领域——关联关系与规范配置

## 文档生命周期关联模式

+   软件领域的文档按研发过程形成纵向关联链，编号反映生命周期阶段：

    | 阶段编号 | 文档类型     | 对应编写规范 | 典型上游         | 关联性质 |
    | -------- | ------------ | ------------ | ---------------- | -------- |
    | 01       | 战略规划     | S01          | -                | -        |
    | 02       | 宏观需求     | S02          | 01-战略规划      | 上下游   |
    | 03       | 领域设计     | S03          | 02-宏观需求      | 上下游   |
    | 04       | 系统设计     | S04          | 03-领域设计      | 上下游   |
    | 05       | 详细需求     | S05          | 04-系统设计      | 上下游   |
    | 06       | BC总体设计   | S06          | 05-详细需求      | 上下游   |
    | 07       | BC详细设计   | S07          | 06-BC总体设计    | 上下游   |

## 关联规则

+   **纵向关联（上下游）**：编号小的阶段是编号大的阶段的上游。上游文档完成后，作为下游文档的隐性约束
+   **横向关联（并行）**：同一阶段编号下，不同模块/BC的文档互为并行关联。并行文档之间须保持术语、接口、协议的一致性
+   **总分关联**：上一阶段的文档与下一阶段的多份文档之间形成总分关系。如 04-系统设计 是多份 05-详细需求（各模块）的上级文档
+   项目级 CLAUDE.md 中的具体关联关系表以本模式为基础，按项目实际情况细化

## 编写规范对应关系

+   每种文档类型有对应的编写规范（S系列），在 specs/software/phase-specs/ 目录下
+   编写/修订前通过 `/load-spec` 手动加载目标文档类型对应的规范
+   **S10-软件文档体例规范**是 S01-S07 的元规范，约束 S01-S07 自身的编写方式（如章节定义结构、参考模板格式等）。S10 仅在编写/修订 S01-S07 规范文档本身时加载，不在编写业务文档时加载
+   S08-S09 为预留编号，暂未使用

## 领域级背景约束

+   specs/software/context/ 下的文档（如 N00-软件研发过程规范）定义了软件研发的整体流程和阶段划分
+   这些背景约束通过 `/load-context` 手动加载，为所有软件领域文档提供过程框架
