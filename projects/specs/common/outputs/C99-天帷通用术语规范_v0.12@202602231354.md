# 2 术语使用规则

## 2.1 单一来源真理

+   天帷通用术语统一在本文档定义，各文档不再重复定义。
+   各文档引用本规范中已定义的天帷通用术语时，只需指向《C99-天帷通用术语规范》即可，不需要标注具体章节编号。
+   各文档仅在自身附录中定义其特有的、本规范未覆盖的文档专有术语。

## 2.2 中英文对照规则

+   首次出现时使用"中文术语 (English Term)"格式，如"限界上下文 (Bounded Context, BC)"。
+   后续出现时仅使用中文术语，如"限界上下文"。
+   常用术语可使用约定简称（如BC、AC、NFR），首次应标注完整形式。
+   代码/配置中使用英文形式或编号前缀，如BC12、EPIC001。
+   技术领域通用的英文缩写可直接使用（如API、HTTP、JSON、DDD）。

## 2.3 复合术语构成规则

+   优先使用标准术语组合，避免自创术语。
+   复合术语遵循"修饰词 + 核心术语"结构。
+   示例：用户登录API接口（修饰词：用户登录，核心术语：API接口）；租户生命周期管理（修饰词：租户，核心术语：生命周期管理）。

## 2.4 编号前缀使用规则

+   编号前缀（如EPIC、BS、US、AC等）是技术标识符，在代码和文档编号中使用英文。
+   编号前缀是《P01-软件开发编号规范》定义的标准标识，不需要"翻译"。
+   文档正文描述概念时使用中文术语（如"史诗"、"业务环节"、"用户故事"）。
+   引用具体条目时使用编号（如EPIC001、BS-USER-onboard）。

# 3 天帷通用术语定义

+   以下术语为天帷体系自定义或赋予特定含义的跨文档通用术语。AI评审和编写文档时应严格按此定义使用。
+   未在本文档中定义的业界标准术语，按其业界通用含义理解。
+   业务价值生态 (Business Value Ecosystem)：产品所处的业务生态系统全景，是推导业务域、实体、利益相关方的共同源头。
+   业务域 (Business Domain, DOM)：系统按业务职能划分的顶层分区，业务环节的领域分类，按业务价值链或职能域划分。
+   业务环节 (Business Segment, BS)：客观发生的一个或一组业务节点，不依赖系统是否存在；描述业务域内"在什么情境下发生什么"，是活动线的最细粒度。
+   工作场景 (Work Scenario, WS)：角色进入系统的切入点，描述角色在业务中承担的职责域；分为直接作业场景（关联业务环节）和间接作业场景（管理/分析/协调类）。
+   关键实体 (Key Entity, EN)：价值生态中的具体核心业务对象，具有明确的业务含义和归属，具有明确生命周期状态流转。
+   困境 (Dilemma, DLM)：业务层面达成目标的系统性障碍，描述"为什么目标难以实现"，是宏观需求分析的输入之一。
+   痛点 (Pain Point, PP)：角色在工作中面临的具体问题，描述"角色在实际工作中遇到了什么困难"，是Epic推导的直接输入。
+   期望 (Expectation, EXP)：系统角色对理想工作状态的正向描述，与痛点互为补充，共同作为Epic推导的输入。
+   关键旅程触点 (Key Journey Touchpoint, TP)：角色在完成业务目标的旅程中与系统或业务流程发生交互的关键节点，是工作场景（WS）的进一步细化。
+   业务验收场景 (Business Acceptance Scenario, BAS)：从业务视角定义史诗（Epic）"完成"的端到端验收场景，是史诗级别的业务验收依据。
+   特定利益相关方 (Specific Stakeholder, SH)：从利益相关方中识别的具体人员群体或组织，按组织维度定位（角色线CL1层级）。
+   系统角色 (System Role, SROLE)：按业务职能对系统使用者进行分类的身份标签，携带预设权限集合，独立于GRC制度内容定义。
+   业务角色 (Business Role, BROLE)：GRC制度文档中显式定义的职责承担者，明确在特定合规活动或管控流程中谁负责哪个环节，独立于平台权限体系存在。
+   语义ID (Semantic ID)：条目的正式标识符，由类型前缀和语义片段组成，表达业务含义，用于系统存储、代码引用和文档索引。
+   短号 (Short Code)：条目的便捷引用标识符，由类型前缀和递增数字组成，无业务含义，用于日常交流和快速检索。
+   双轨标识 (Dual-Track Identification)：同一条目同时拥有语义ID和短号两种标识，两者保持一一对应关系。
+   类型前缀 (Type Prefix)：语义ID和短号的首段，标识条目所属类型，如EPIC、US、API等。
+   语义片段 (Semantic Segment)：语义ID中类型前缀之后的各组成部分，表达业务含义，如业务域、对象、动作、场景等。
+   三线演绎 (Three-Line Deduction)：宏观需求分析方法论，通过活动线、角色线、实体线三条分析线推导和验证Epic。
+   抽象过滤 (Abstraction Filtering)：Epic推导过程中将候选需求按粒度分类的方法，Epic级保留，Feature级降级为候选项。
+   宏观需求 (Macro Requirements)：产品全局视角的需求，定义Epic级业务能力，不涉及具体交付。
+   基础性需求 (Foundation Requirements, FR)：跨限界上下文(BC)的通用能力需求，如认证、权限、审计等功能性横切关注点。
+   需求域 (Requirements Domain)：按需求的抽象层级和关注视角划分的需求分类维度：业务需求域（StRS）、系统架构需求域（SyRS）、软件开发需求域（SRS）。
+   大BC (Large Bounded Context)：涉及多个子领域的复杂限界上下文，总体设计阶段需进行子领域划分。
+   小BC (Small Bounded Context)：业务范围单一、无需子领域划分的限界上下文。
+   BC总体设计 (BC High-Level Design)：BC层面的高层技术方案设计，确定整体架构、模块划分和关键技术选型。
+   BC详细设计 (BC Detailed Design)：BC或子领域层面的细粒度技术设计，包括数据模型、接口定义、核心算法等。
+   条目 (Item)：软件开发过程中需要唯一标识和追溯的工作产物或管理对象。
+   知识单点定义原则：每个知识点只在一个位置完整定义，其他位置通过精确引用（具体章节号）指向该定义位置。

# 4 易混淆术语区分

+   以下术语对在不同语境中含义不同，文档中应明确指代。
+   业务域 vs 限界上下文：业务域(DOM)是编号体系中的顶层分区标识；限界上下文(BC)是DDD中的业务边界划分单元。一个业务域下可包含一个或多个BC。
+   业务域 vs 领域(DDD)：业务域是编号体系概念；领域(Domain)是DDD的分析概念，用于问题空间划分（含核心域、支撑域、通用域）。
+   事件 vs 领域事件：事件(EVT)是宏观需求语境中触发实体生命周期状态流转的动作；领域事件(Domain Event)是DDD设计语境中限界上下文之间的通信机制。
+   关键实体 vs 实体(DDD)：关键实体(EN)是宏观需求分析中识别的核心业务对象，侧重业务建模；实体(Entity)是DDD详细设计中的领域对象概念，侧重技术建模。
+   系统角色 vs 业务角色：系统角色(SROLE)是按业务职能对系统使用者分类的身份标签，携带预设权限；业务角色(BROLE)是GRC制度中定义的职责承担者。
+   宏观需求 vs 详细需求：宏观需求是产品全局视角的Epic级业务能力定义；详细需求是BC内的Feature/User Story/AC级实现定义。
