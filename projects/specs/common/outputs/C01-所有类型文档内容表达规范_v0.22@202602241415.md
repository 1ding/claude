# 1 总则

+   文档编号：C01
+   文档名称：内容表达规范
+   适用对象：所有由AI协助编写或优化的文档
+   管辖范围：语言写法、逻辑性与段落结构、用语风格、术语处理、信息表达形式选择（表格/列表/图）
+   不管辖的范围：
    -   文档分区结构、总则编写要求、附录体系、修订记录格式、密级标注（见《C11-文档通用结构规范》）
    -   排版格式（见《C02-排版格式规范》）
    -   规范类文档的内容组织结构（见《M01-通用规范编写规范》）
    -   项目文档的大纲生成与内容生成（见各自专用规范）
+   文档密级：【商密二级】 -- 外发必追责。

# 2 知识单点定义原则

+   知识单点定义原则：每个知识点只在一个位置完整定义，其他位置通过精确引用（具体章节号）指向该定义位置（见《C99-天帷通用术语规范》） 。
+   冗余判断方法：如果删掉某段话后，该知识点在文档中仍有完整定义存在，则该段是冗余的，应改为引用。
+   引用格式："见X.X节"或"按X.X节定义"。

# 3 术语处理规则

+   术语分为三层，按作用域从大到小依次为：业界通用术语、天帷通用术语、文档专有术语。三层术语均在附录中集中管理，正文中不单独设术语章节。
+   业界通用术语：软件工程及相关领域的业界主流术语（如 MECE、MVP、CI/CD、UAT、SRS、UML、敏捷、瀑布、微服务、RESTful、ORM、TDD、代码审查、持续集成、回归测试等）。AI 模型对这些术语已具备充分的理解能力，因此不需要定义，仅在附录中列出术语名称即可。列出的目的是明确"本文档使用了哪些业界术语"，便于术语审计和一致性检查。
+   天帷通用术语：天帷体系内跨文档共用的术语（需求层级、业务建模、编号体系、过程方法等），由《C99-天帷通用术语规范》统一定义。所有对天帷通用术语的引用（包括正文中的引用和附录中的引用）只需指向《C99-天帷通用术语规范》即可，不需要标注具体章节编号。各文档在附录中引用天帷通用术语时，仅列出术语名称和来源，不重复定义。
+   文档专有术语：某份文档特有的、上述两层均未覆盖的专业术语，在该文档的附录中完整定义。
+   三层术语的判定顺序：遇到一个术语时，先判断是否属于业界通用术语；若不是，再判断是否属于天帷通用术语；若仍不是，则归入文档专有术语并在附录中定义。
+   正文中的术语处理：含义偏离业界常见用法的术语，或天帷体系赋予了特定含义的术语，在正文中首次出现时内联定义。内联定义格式示例："业务活动（即PL2流程节点）应满足以下条件…"。内联定义用于帮助读者就地理解，不替代附录中的完整定义。

# 4 逻辑性与段落结构要求

+   信息呈现应遵循逻辑顺序，具体包括以下九条要求。
+   单个一级标题下的内容不应超过 16K tokens（默认值，用户可在编写指令中覆盖）。超过时应将该标题拆分为多个一级标题，使每个一级标题对应一个独立的语义单元。此要求确保文档入知识库后，AI 能有效理解每个章节。每章除此容量上限外，不设置其他默认限制（不限制标题层级深度、目录层级、行数等）。
+   一个段落只承载一个判断。段落中出现"但是"、"另外"、"同时还需要注意"时，应拆段或拆为独立条目。
+   先建立前提再展开细节。先给出总体原则或分类框架，再逐项展开。错误做法：细节规则散落各处，读者需要自行归纳总体框架。
+   同一分类维度下的条目应满足MECE原则（互不重叠、合在一起穷尽）。列举分类时，各类别之间不应有交叉，合在一起应覆盖全部情况。
+   因果关系显式表达。如果A是B的原因或前提，应用"因此""→"等连接词显式标注，不依赖读者自行推断。
+   递进关系按方向一致排列。如果多条规则存在严格程度的递进（如"通用→严格→宽松"或"必须→推荐→允许"），应按递进方向连续排列，不交叉穿插。
+   并列条目按统一维度分组。如果多条规则本质上属于同一类别（如都是"具体化"要求或都是"精简化"要求），应显式分组，不混杂排列。
+   规则聚合原则。针对同一对象的多条规则，应按该对象的某一方面或维度聚合为一个规则组。每个规则组围绕一个明确主题，形成自包含的整体，使读者阅读单个规则组即可理解该方面的全部要求。错误做法：将不同方面的规则交叉散落在多个位置，读者需跨章节拼凑才能获得完整约束。
+   规则组的展开结构。一个规则组内部应按"规则要求 → 示例（可选） → 补充说明（可选）"的顺序展开。规则要求是核心规则，应首先给出；示例用于帮助理解，紧跟其所例证的要求；补充说明用于交代边界条件、例外情况或设计原因，置于末尾。

# 5 表格与列表的选择原则

+   原则上优先使用列表，仅在表格比列表更易于理解时才使用表格。
+   使用表格的场景：需要同时对齐三个及以上字段进行多维度对比时（如"类别-定义-缺失后果"三列对齐），列表无法表达列与列之间的对齐关系。
+   使用列表的场景：单纯的"类别-定义"映射、正反例对照、枚举项说明，每条自包含一个完整语义，用列表逐条陈述更清晰。
+   短表格（≤4行）可保留表格形式，语义集中，影响小。长表格（≥5行）应优先考虑改写为逐条列表，避免语义稀释。

# 6 图的使用原则与视图归属

+   图（流程图、架构图、关系图等）用于表达多要素之间的空间关系、流转关系或层级关系，文字和表格无法等效替代时才使用图。
+   判断标准：如果去掉这张图，读者是否会理解错要素之间的关系？会→用图；不会→用文字或表格。
+   图应自包含：图中应包含足够的标注，使读者无需阅读正文即可理解图的核心含义。
+   文字自足原则：正文的文字描述应确保AI不看图也能完整理解所有规则、约束和要素关系。图是文字的可视化增强，不是信息的唯一载体。如果某个关键信息只存在于图中而文字中没有，则该信息在知识库检索时将丢失。
+   视图归属原则：视图跟着其表达的内容走，紧跟其所描述的主题，不单独集中放置。
+   必要视图留在正文：表达正文核心要素之间的关联关系。典型场景为要素关联图、核心流程图、要素分类图、业务流程图、用例图、架构图。
+   辅助视图归入附录或资料性章节：表达方法论推导或概念教学。典型场景为演绎过程图、概念解释图、推导逻辑链图。

# 7 语言精简要求

+   正文用语应具体、紧凑、低认知负担。具体分为"具体化""精简化""清晰化"三组要求。
+   具体化要求——用具体信息替代模糊表述：
+   用数据代替修饰词。正确写法："处理速度为 X ms/batch"。错误写法："能够非常高效地处理"。
+   名词具体不抽象。正确写法："对 embedding 向量做 L2 归一化"。错误写法："对相关数据进行处理"。
+   精简化要求——删除不增加信息量的文字：
+   砍万能动词包装。正确写法："清洗数据"。错误写法："对数据进行清洗"。
+   砍正确的废话。正确写法："使用前配置 API Key"。错误写法："在使用该接口之前，需要先进行相关的配置工作"。
+   控制"的"字链（≤2个）。正确写法："Transformer 预训练语言模型的推理性能"。错误写法："基于 Transformer 的预训练的语言模型的推理的性能"。
+   清晰化要求——换用更易理解的表达方式：
+   否定句改肯定句。正确写法："先完成初始化，再调用该方法"。错误写法："不要在未完成初始化的情况下调用该方法"。

# 8 语言风格要求

+   使用陈述句，避免感叹句。
+   使用主动语态、短句优先，必要时使用被动语态。警惕长定语从句嵌套。
+   一个概念全文只用一个术语，不在同义词之间交替使用。
+   避免口语化、主观性或情感化表达。

# 9 规范性用语定义

+   规范性用语用于表达规则的强制程度，所有文档应统一使用以下约定。
+   规范性用语"应"表示强制要求。
+   规范性用语"宜"表示推荐做法。
+   规范性用语"可"表示允许的选项。
+   规范性用语"不应"表示强制禁止。
+   规范性用语"不宜"表示不推荐。
